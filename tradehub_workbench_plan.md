# TradeHub Workbench — 全景规划与开发路线图

## 一、产品定位

TradeHub Workbench 是面向有色金属散货贸易的**一人公司全流程工作台**。
从报价测算到结算收款，从货物追踪到敞口管理，所有电脑前能完成的工作在一个应用内闭环。

核心用户画像：2-3人团队（甚至1人公司），同时跟5-8个合同，采购和销售两端都要管，
需要随时知道"钱在哪、货在哪、下一步做什么"。

---

## 二、业务全景：一个合同的完整生命周期

```
报价测算          合同签订          收发货           定价             结算            资金收付
───●──────────────●──────────────●──────────────●──────────────●──────────────●───
   │              │              │              │              │              │
   │ 对方报价     │ 上传合同     │ 磅单录入     │ 均价计算     │ 核对计算     │ 预付款
   │ 快速估算     │ LLM解析      │ 化验单录入   │ 点价锁定     │ 生成结算单   │ 尾款
   │ 有利润→谈    │ 人工确认     │ 配对匹配     │ 人工确认     │ 发送对方     │ 费用结清
   │              │              │              │              │              │
   │  测算可留存   │              │              │              │              │
   └──→签约后转为合同,配方微调后直接用──────────────────────────────┘              │
                                                                                │
                  ╔══════════════════════════════════════════════════════════════╗
                  ║  全程伴随：库存追踪 · 资金头寸 · 敞口监控 · 行情关注        ║
                  ╚══════════════════════════════════════════════════════════════╝
```

**关键洞察：测算 → 合同是有继承关系的。** 测算阶段用假设数据验证的计价配方，
签约后可以直接转为合同的正式配方（微调参数即可），不需要重新录入。

### 3.1 页面总览

```
┌─────────────────────────────────────────────────────┐

├─────────┬───────────────────────────────────────────┤
│ 侧边栏   │  主内容区                                │

```

### 3.2 各页面职责

| 页面 | 核心问题 | 主要操作 | 优先级 |
|------|---------|---------|--------|
| **驾驶舱** | 现在该做什么？ | 查看待办、资金快照、库存快照、行情 | P0 |
| **合同工作流** | 这个合同进展到哪了？ | 测算/解析/跟单/定价/结算/资金 6个阶段 | P0 |
| **资金总览** | 钱够不够？谁欠我、我欠谁？ | 应收应付、费用明细、利润估算 | P1 |
| **库存总览** | 货在哪？敞口多大？ | 库存明细、出入库、加权成本、敞口 | P2 |
| **行情中心** | 今天什么价？影响哪些合同？ | 实时行情、待确认基准价队列 | P2 |

### 3.3 合同工作流 — 6个阶段面板

这是最核心的页面。打开一个合同后，顶部显示阶段进度条，
系统根据数据状态自动定位到当前阶段，也可以手动切换回看。

```
  测算  ──→  解析  ──→  跟单  ──→  定价  ──→  结算  ──→  资金
  [评估]    [确认]    [采集]    [锁价]    [核对]    [收付]
```

#### 阶段 0：测算（报价评估）

用途：对方给了报价条件，快速算有没有利润。

布局：
- 左：输入面板（假设品位、假设重量、对方的计价条件）
- 右：实时计算结果（预估金额、预估利润）
- 底部：保存测算 / 转为合同

特点：
- 数据全部可手动输入，不依赖真实单据
- 计价配方可以手动构建，也可以粘贴合同条款让 LLM 解析
- 保存后留存为"测算记录"
- 签约后一键"转为合同"，配方和参数继承过去，状态变为正式

#### 阶段 1：解析（合同确认）

用途：合同签了，确认系统理解的计价规则是否正确。

布局（双栏对照）：
- 左：合同原文（PDF渲染 / OCR文本 / 手动粘贴）
- 右：结构化摘录（计价配方列表，可编辑）
- 点击右栏某个元素 → 左栏对应条款高亮

操作：逐条核对，修改，确认锁定。

#### 阶段 2：跟单（数据采集）

用途：磅单和化验单陆续到，录入并配对。

布局：
- 上方：进度条（磅单 N/M 张，化验单 K 份，未配对 X 个）
- 主区域：按样号分组的配对视图
  - 每组显示：样号、磅单列表（车号/净重/日期）、化验单数据（品位/水份）
  - 未配对的标红
- 右侧或底部：快速录入入口（拍照上传 / 手动填写）

#### 阶段 3：定价（基准价确认）

用途：确认每个元素的基准价。

布局：
- 元素列表，每个元素一个卡片：
  - 来源（均价/点价）
  - 锚定规则（从合同继承）
  - 当前状态（待确认/已锁定）
  - 价格值（可编辑/可触发自动抓取）
  - 覆盖磅单范围

操作：确认均价 / 录入点价 / 锁定。

#### 阶段 4：结算（计算验证）

用途：所有数据齐了，核对计算结果。

布局：
- 上方：就绪检查（✓磅单 ✓化验单 ✓基准价，全绿才能算）
- 主区域：按样号分组的计算明细
  - 每个样号 × 每个元素：展开显示数量管道和单价管道的逐步过程
- 右侧：汇总面板（元素货款合计、扣杂合计、费用、净额）
- 底部：生成结算单（导出Excel / 推送飞书 / 生成PDF）

#### 阶段 5：资金（收付与费用）

用途：管理这个合同的所有资金往来。

布局：
- 资金流水列表（预付款/结算款/尾款/费用，每笔有日期/金额/方向/摘要）
- 费用明细（运费/化验费/差旅费，可关联到具体批次）
- 底部汇总：应付总额、已付金额、余额（正=还欠/负=多付）
- 利润估算（仅销售合同或采销联动时）：收入 - 成本 - 费用 = 毛利

## 四、数据模型（原型阶段）

原型阶段脱离飞书，用本地 JSON/SQLite 存储。数据模型与 Bitable 表结构对齐，
未来迁移时字段名直接映射。

### 4.1 核心表

| 表名 | 职责 | 行粒度 |
|------|------|--------|
| contracts | 合同基本信息 | 1行/合同 |
| pricing_rules | 计价配方（recipe JSON） | N行/合同（每元素+每扣杂1行） |
| base_prices | 基准价 | N行/合同（元素×磅单组） |
| weigh_tickets | 磅单 | N行/合同 |
| assay_reports | 化验单 | N行/合同（按样号） |
| settlement_items | 结算明细（审计底稿） | 样号×元素 行 |
| cash_flows | 资金流水 | 每笔资金动作1行 |
| expenses | 费用明细 | 每笔费用1行 |
| estimates | 测算记录 | 1行/测算，含快照数据 |

### 4.2 未来扩展表（P2）

| 表名 | 职责 |
|------|------|
| inventory | 库存明细（按批次/仓库） |
| inventory_moves | 出入库记录 |
| goods_allocation | 货物流向（采购→库存→销售） |
| market_prices | 行情数据（每日/每品种） |

### 4.3 计价配方 Schema（Recipe）

```yaml
element: Cu
type: element | deduction      # 计价元素 or 扣杂
clause_text: "原文摘录..."     # 人读的
quantity:
  type: wet_weight | dry_weight | metal_content
  gradeField: cu_pct           # 品位字段名（metal_content时必填）
  gradeDeduction: 1.0          # 品位扣减（可选）
  moistureCap: 0.10            # 水份超标阈值（可选）
unitPrice:
  base: 9800 | ref:base_price  # 固定数值 or 引用基准价表
  unit: 元/金属吨 | 元/干吨 | 元/吨 | 元/克
  operations:                  # 原子操作链
    - type: tier_multiply | segment_accumulate | tier_fixed | base_subtract_tier | grade_linear_adjust
      lookupBy: base_price | cu_pct | au_gt | ...
      tiers: [...]
      anchor: 18.0
      segments: [...]
minGrade: 13.0                 # 低于此不计价（可选）
maxGrade: null                 # 高于此另议（可选）
```

## 五、技术架构（原型阶段）

```
┌─────────────────────────────────────────┐
│  前端：React SPA                         │
│  路由：React Router                      │
│  状态：Zustand（轻量）                    │
│  计算：JS 版 Recipe Evaluator（前端实时） │
│  UI：Tailwind + 自定义组件               │
└──────────────────┬──────────────────────┘
                   │ REST API (JSON)
┌──────────────────┴──────────────────────┐
│  后端：FastAPI (Python)                  │
│  计算：Python 版 Recipe Evaluator（Decimal）│
│  存储：SQLite（开发期）→ Bitable（生产期）│
│  AI：本地 Qwen3 32B（合同解析）           │
│  OCR：GLM-OCR API（磅单/化验单识别）      │
└─────────────────────────────────────────┘
```

**前后端计算引擎保持一致**：
- 前端 JS 版用于交互页面实时预览（拖滑块即时响应）
- 后端 Python 版用于正式结算（Decimal 精度，写入存储）
- 两者共享同一个 Recipe Schema，用 fixture 交叉验证一致性

**存储层抽象**：
- 定义 DataStore 接口（get/list/create/update）
- 原型期实现 JsonFileStore（读写本地 JSON 文件）
- 生产期实现 BitableStore（调用飞书 API）
- 切换只改配置，不改业务代码

## 八、设计原则

1. **阶段驱动，不是功能驱动**：用户看到的是"这个合同到哪一步了"，不是"点磅单tab录入"。
2. **计算即时反馈**：任何数据变动，结算预览立刻更新。不需要"点击计算"按钮。
3. **配方可审计**：每个计价元素的配方对应合同原文，修改有记录，LLM草稿和人工确认有区分。
4. **测算可留存，可转化**：报价阶段的工作不浪费，签约后直接继承。
5. **渐进增强**：先跑通结算（核心价值），再扩展跟单/资金/库存。
6. **存储解耦**：原型用本地文件，随时可切飞书Bitable，业务代码不改。
