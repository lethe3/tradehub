## 一、阶段目标

从"单据录入工具"升级为"跟单计算引擎"。

阶段一验证了：飞书 Bot 收消息 → OCR 提取 → 人工确认 → 写入 Bitable。
阶段二要跑通：合同 + 磅单 + 化验单 → 数据串联 → 结算计算 → 资金流水 → 自动生成结算单。

核心衡量标准：**用一组真实业务数据（一份合同 + 若干磅单 + 化验单），从录入到生成结算单全流程自动化。**

## 二、数据架构（五表 + 一视图）

### 表关系总览

```
合同表（源头）
 ├─→ 磅单表      1:N（关联合同）
 ├─→ 化验单表    1:N（关联合同），化验单↔磅单通过样号串联

结算单 = 视图（按合同筛选流水表 → 渲染输出）
```

## 三、计价模型（两轴分离）

### 轴一：基准价确定方式

解决"参考什么价格"的问题，输出一个数字。

| 方式 | 输入 | 逻辑 |
|------|------|------|
| 均价 | 数据源+代码+起止日期+交易日/自然日 | 拉取价格序列 → 求均值 |
| 点价 | 数据源+代码+点价日期+确认价格 | 直接使用确认价 |
| 合同固定 | 合同约定数字 | 直接使用 |

> 阶段二先支持"合同固定基准价"，均价和点价留接口不实现。

### 轴二：计价公式 f(品位, 基准价) → 单价

每个计价元素各有一套独立的公式。常见模式：

| 模式 | 公式结构 | 示例 |
|------|----------|------|
| 固定价 | 单价 = 常数 | 单价 = 17250 元/金属吨 |
| 系数 | 单价 = 基准价 × 系数% | 单价 = 基准价 × 75% |
| 扣减 | 单价 = 基准价 - 固定值 | 单价 = 基准价 - 400 元/干吨 |
| 品位调价 | 基准价 ± (品位差 × 调价系数) | 品位偏离基准每1%，±20元 |
| 条件分段 | IFS(品位≥阈值1, 公式1, …) | ≥5%计90%系数，≥1%计80% |
| 混合 | 以上的组合 | 基准价×系数 - 扣减 + 品位调价 |

> 注意单价单位：元/吨、元/干吨、元/金属吨 指向不同的计算基数。

### 杂质扣款（独立于计价公式）

从总金额中扣减，不影响单价。

| 模式 | 逻辑 |
|------|------|
| 简单阶梯 | 超标量 × 扣款单价/干吨 |
| 累进阶梯 | 类似超额累进税：每段各自计算再汇总 |
| 取整策略 | 按比例(1.47→1.47) vs 向上取整(1.47→1.5) |
| 买卖方有利 | 采购方通常选择对己有利的取整方式 |

## 五、代码模块设计

### 新增模块

```
core/
├── linking.py          ← 数据串联引擎
│   ├── match_by_sample_id()   样号匹配：化验单↔磅单
│   └── build_batch_view()     构建完整批次视图
│
├── settlement.py       ← 结算计算引擎
│   ├── calc_dry_weight()      干重 = (净重-扣重) × (1-水分%)
│   ├── calc_metal_quantity()  金属量 = 干重 × 品位%
│   ├── calc_unit_price()      单价 = f(品位, 基准价) ← 可插拔公式
│   ├── calc_deductions()      杂质扣款（阶梯累进）
│   └── generate_cash_flows()  输出：资金流水记录列表
│
└── models/
    ├── batch.py        ← BatchView: 完整批次数据模型
    ├── pricing.py      ← PricingRule: 计价公式（多态）
    └── cash_flow.py    ← CashFlowRecord: 流水记录
```

### 架构约束

- `core/` 只依赖 Pydantic 模型，不 import `feishu/` 或 `ai/`
- 所有计算函数：纯输入 → 纯输出，可用 mock 数据做充分单元测试
- Bitable 读写由 `feishu/` 层完成，结果传入 `core/` 层函数

## 八、从工作流程文档提取的校验规则（待录入 Work/Errors/）

| 错误编号 | 类型 | 规则 |
|----------|------|------|
| E01 | 单位不一致 | 金属量单位(g vs kg)影响小数位→最终金额偏差 |
| E02 | 取整分歧 | 按比例(1.47→1.47) vs 向上取整(1.47→1.5)，需在合同中明确 |
| E03 | 品位截断 | 品位52.64截断为52→调价金额偏差 |
| E04 | 分段遗漏 | 品位跨多段阈值时需累进计算，不能只算最后一段 |
| E05 | 日期错误 | 基准价对应错误日期 |
| E06 | 条件顺序 | IFS条件应从高到低输入 |
| E07 | 扣款遗漏 | 扣款金额未从总价中减去 |
| E08 | 计价遗忘 | 合同约定的计价元素未计算 |
| E09 | 金额精度 | 金额保留两位小数 |

> 这些错误将成为 core/settlement.py 的校验断言和警告提示。
