# TradeHub 开发教训

## 2026-03-14

### 事件

开发通用 Bitable 框架时，字段类型转换逻辑错误，导致测试多次失败。

---

### 问题本质

| 表面现象 | 本质问题 |
|---------|---------|
| 单选字段传了选项 ID | **我没有验证代码假设**——以为 SDK 需要 ID，实际需要名称 |
| 关联字段格式错误 | **我没有查阅文档**——SKILL 明确说传字符串数组 |
| 多次尝试仍失败 | **我没有及时求助**——应该直接问用户而不是自己硬扛 |
| 不知道字段类型变了 | **我没有确认现状**——用户改了单向关联为双向关联，但我没问 |

---

### 根因分析

1. **急于求成**：没有先读文档再动手，直接根据旧经验写代码
2. **路径不熟**：不知道 SKILL 文档在哪里，没有第一时间查找
3. **闭门造车**：失败后继续自己尝试，没有问用户确认
4. **假设代替确认**：假设 SDK 行为，没有去验证

---

### 改进措施

- [ ] 开发前必须先查 SKILL 文档
- [ ] 不确定的字段类型先问用户确认
- [ ] 遇到 3 次以上失败，主动求助而不是继续硬扛
- [ ] 更新 CLAUDE.md 添加飞书开发规范

---

### _commit 流程教训_

**问题**：任务完成后直接 commit，没有先让用户确认需要修改的文件

**正确流程**：
1. 列出代码变更和需要用户确认的 Obsidian 文件
2. 明确标注哪些是你负责修改的（如 status.md、log.md 的草稿部分）
3. 等你说"可以了"后再 commit
4. 禁止在用户确认前 commit
